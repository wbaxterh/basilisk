Section 1

1. Executive Summary

Basilisk is an all-in-one analytics, portfolio-tracking, and trading platform for the Cardano blockchain. It targets the same job-to-be-done as TapTools — today's market leader — while differentiating on speed, price transparency, real-time data, trader-grade tooling, and an open developer API.

The name nods to the basilisk — the serpent-king of myth — keeping the Cardano "snek" lineage while evoking Roko's Basilisk, the AI thought experiment: a fitting emblem for an agent-native platform. The product spans three surfaces:

  • Web app — charts, portfolio, wallet profiling, market screeners, DeFi/NFT analytics, and in-app swaps.
  • Public Data API — a paid REST + WebSocket API for developers, bots, and integrators.
  • Mobile app + alerts — portfolio tracking, price/whale alerts, and push notifications.

Crucially, the API is agent-native: pay-per-call via the x402 micropayment protocol (no keys, no subscription) and queryable by LLM agents through an MCP server — turning Basilisk into the data and execution backbone for agent-to-agent (A2A) trading on Cardano (see §8).

The Opportunity

TapTools is dominant but carries known friction: an NFT-gated subscription model that confuses non-crypto-native users, occasional data latency, and an API priced for enterprises. Basilisk can win the prosumer trader + builder segment with cleaner UX, real-time streaming data, transparent fiat pricing, and a generous free developer tier.

2.2 TapTools Business Model

  • PRO is NFT-gated. "Tappy" NFTs grant access — e.g. a 4,000-supply sale at 420 ADA each; lifetime via 5 original Tappy NFTs; annual via a single Tappy02 NFT. Secondary-market pricing floats after the primary sale.
  • Day passes: ~10 ADA for a 1-day PRO pass.
  • API: separate developer subscription tiers; positioned as one of the most in-depth API services in the ecosystem.
  • Quality thresholds: Top 50 requires ≥1M ADA lifetime volume; Top Gainers requires ≥5,000 ADA LP; NFT portfolio requires ≥5,000 ADA lifetime volume.

3. Goals & Non-Goals

3.1 Goals

  1. Achieve feature parity with TapTools' core analytics + portfolio surfaces by MVP.
  2. Differentiate on real-time WebSocket data, fiat-transparent pricing, and trader UX.
  3. Ship a developer API with a free tier to seed an ecosystem of bots and integrations.
  4. Deliver sub-second chart loads and <5s on-chain data freshness.
  5. Build a sustainable revenue model: subscriptions + API + optional swap fees.

3.2 Non-Goals (v1)

×Building our own DEX or AMM — we aggregate, not operate liquidity.
×Cross-chain support beyond Cardano (future consideration).
×Custodial wallet / fund custody — Basilisk is non-custodial, read-only + sign-via-wallet.
×A full tax-reporting engine (post-MVP integration / partner).

7.4 Suggested Stack

Frontend: Next.js (web) + React Native/Expo (mobile); TradingView Charting Library / lightweight-charts.  Backend: TypeScript (NestJS) or Go for perf-critical ingestion; Rust for DEX decoders if needed.  Data: Postgres, TimescaleDB/ClickHouse, Redis.  Infra: Kubernetes; the node + indexer is the main ops cost.

Key Technical Risks

DEX decoder maintenance — protocols upgrade scripts; adapters need active upkeep. Data accuracy — low-liquidity tokens produce noisy prices, needing robust outlier handling. Node/indexer ops — mitigate early by leaning on Blockfrost/Koios while building in-house. Scale — real-time fan-out to many WebSocket clients.

Section 8

8. The Agent Layer — A2A Trading & x402

AI agents are becoming first-class market participants on Cardano. They need the same three things humans get from Basilisk — data, discovery, and execution — but in machine-native form: programmatic, pay-per-use, and trustless. The Agent Layer turns Basilisk into the data and execution backbone for agent-to-agent (A2A) trading, monetized natively via the x402 micropayment protocol.

8.1 Why Agents Need Basilisk

Agents trading on Cardano today are forced through DEX UIs or raw RPC — running their own infra and stitching pricing together by hand. Basilisk already computes the hard parts: VWAP pricing, OHLCV, liquidity, smart-money signals, and aggregated swap routes. The Agent Layer simply exposes that as an agent-native interface. Critically, TapTools' API is subscription- and API-key-gated — friction an autonomous agent can't clear (no human to sign up for a SaaS plan with a credit card). Basilisk removes that gate entirely.

8.2 x402-Native API (Pay-Per-Call)

x402 is an HTTP-native payment standard (built on the dormant HTTP 402 Payment Required status, originated by Coinbase). The flow: an agent hits an endpoint, receives a 402 with payment details, settles a micro-amount on-chain, and retries with proof of payment to receive the data. No account, no API key, no subscription. Basilisk runs an x402 facilitator/paywall in front of the data API so every call self-settles for a fraction of a cent.

Figure 8.1 — x402 Pay-Per-Call Flow
sequenceDiagram participant AG as Trading Agent participant API as Basilisk API participant FAC as x402 Facilitator participant CH as Cardano AG->>API: GET /token/price (no key) API-->>AG: 402 Payment Required
price · pay-to address · nonce AG->>CH: Submit micropayment AG->>API: Retry + payment proof API->>FAC: Verify settlement FAC-->>API: Confirmed API-->>AG: 200 OK · VWAP + OHLCV + liquidity Note over AG,API: No signup · no API key · usage-perfect billing
Agents pay only for the calls they make — zero onboarding, no idle subscription cost.

This lets a three-tier access model coexist, each serving a different consumer:

TierWhoMechanism
FreeHobbyists, evaluationRate-limited public endpoints, no payment.
x402 pay-per-call Agent-nativeAutonomous agents & botsMicropayment per request; no signup, no key — settles on-chain.
SubscriptionHumans, teams, enterprisesFlat monthly fiat/ADA rate with API key; predictable cost at volume.

8.3 Agent-to-Agent (A2A) Trading Rails

With pricing and execution exposed, Basilisk becomes the layer agents use to trade with each other:

  • Price discovery — real-time VWAP, liquidity depth, and smart-money signals via the x402 API / MCP server.
  • Execution — the Phase 2 swap aggregator is callable programmatically; agents route and settle on-chain at best price.
  • Optional A2A quote layer — market-maker agents post buy/sell quotes (tighter than pool spreads) that other agents or humans take, settled trustlessly via a Cardano Plutus escrow contract. Cardano's deterministic, no-public-mempool model means no sandwich/MEV extraction by construction.

(tighter than pool) end CP->>BOOK: Take sell quote CP->>ESC: Lock payment MM->>ESC: Lock tokens ESC->>ESC: Atomic settle — no MEV ESC-->>CP: Receive tokens ESC-->>MM: Receive ADA + spread
Basilisk is the reference price; settlement is peer-to-peer via escrow — Basilisk never holds funds or operates a pool.

8.4 Agent Identity, Reputation & Safety

  • First-class agent wallets — agent PnL leaderboards (extending smart-money), labeled agent clusters, and per-agent activity feeds.
  • Identity — optional agent handles via ADA Handle ($handle) plus on-chain reputation accrued from settled trades.
  • Safety rails — per-key spend caps, request rate limits, and a stale-price circuit-breaker that pauses an agent if reference data is older than a threshold.

8.5 MCP Server

An MCP server wraps the x402 API so LLM agents (Claude and others) can query Basilisk with natural-language and structured tools — "what's the 24h VWAP and top holders for SNEK?" — and pay per call transparently. This makes Basilisk the default Cardano data source for the agent ecosystem.

Scope Honesty

The x402 API, MCP server, and agent analytics are aligned with our "aggregate, don't operate liquidity" non-goal — Basilisk serves data and routes swaps but never custodies funds. The optional A2A quote layer (8.3) edges toward operating a venue; it is explicitly Phase 3 and opt-in, settles peer-to-peer via escrow (not a pool), and is gated on the open questions in §12.

Section 12

12. Open Questions

  1. Token/NFT: launch a native Basilisk token/NFT (e.g. $BSL) for perks, or stay fiat-only at launch?
  2. Swap routing: build our own aggregator or partner with DexHunter for execution?
  3. Indexer build vs. buy: how long do we lean on Blockfrost/Koios before full in-house indexing?
  4. x402 settlement asset: settle micropayments in ADA, a Cardano-native stablecoin (USDM/iUSD), or USDC on Base for cross-chain agents? Build our own facilitator or adopt an existing x402 facilitator?
  5. A2A venue boundary: does an agent quote/offer book (§8.3) cross our "don't operate liquidity" non-goal — and if so, do we ship it as a separate product?
  6. Data licensing: pursue B2B data feeds (exchanges, CMC/CoinGecko-style) as an early revenue line?
  7. Brand/legal: confirm "Basilisk" name/trademark availability — it's a common mythological term, so check for crypto-name clashes and domain/handle availability.
